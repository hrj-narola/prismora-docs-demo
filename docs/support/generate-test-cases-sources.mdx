# Generate Test Cases: Input Sources

This page explains the common ways Prismora can generate test cases based on the inputs you provide.

:::tip Choose the source that matches your intent
If you want coverage driven by behavior, use requirements/designs.
If you want coverage driven by changes, use code changes.
:::

## Quick comparison

| Source | Best for | What you should provide |
| --- | --- | --- |
| Code Changes | Change-focused testing | Diffs/patches or selected repository changes |
| Requirements | Coverage mapped to specs | Requirements + relevant sections |
| Design Files | Flow/state/UX-aligned tests | Diagrams, flow specs, UI artifacts |
| Describe To | Fast scenario discovery | Clear behavior, edge cases, success criteria |

## From Code Changes

### Use when

- You have a PR/commit set or code updates that need targeted testing.

### What to provide

- The code changes (diff/patch, selected files, or repository context—depending on your setup)

### Expected outcome

- Prismora derives scenarios based on how the code changed and where behavior may shift.

## From Requirements

### Use when

- You want testing coverage driven by functional or non-functional requirements.

### What to provide

- Requirements/spec documents and their relevant sections

### Expected outcome

- Prismora generates test cases mapped to each requirement or requirement group.

## From Design Files

### Use when

- You have design assets (UI/flow/architecture diagrams) that should influence testing.

### What to provide

- Design files relevant to the scope (and any notes your team uses)

### Expected outcome

- Prismora generates tests aligned to flows, states, and expected behaviors in the designs.

## Describe To (Free-form)

### Use when

- You want Prismora to generate tests from your plain-English description.

### What to provide

- A clear description of the feature, expected behavior, edge cases, and success criteria

### Expected outcome

- Prismora converts your description into a structured test set.

## Best practices across all sources

- Provide scope boundaries (what’s in / what’s out).
- Include acceptance criteria or definitions of “correct behavior” when available.
- Start with a smaller number of test cases, then expand once the output quality is confirmed.
